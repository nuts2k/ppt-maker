## 3. 子任务地图

### M0 项目骨架与技术基线

计划子任务：`project-foundation`

执行内容：

- 确定最小目录结构、包管理器、TypeScript/Node 版本和质量命令。
- 调研并验证 macOS 离线 OCR、图片处理、PptxGenJS 和 PowerPoint for Mac 的最小兼容链路。
- 建立 16:9 坐标、工作目录和核心数据契约。
- 使用首批真实代码更新 Trellis backend/frontend 规范，避免继续保留空模板或虚构规则。

验证重点：开发环境可重复安装；最小命令可运行；测试、类型检查和格式检查可执行；`open-design/` 无修改。

### M1 单页端到端原型

计划子任务：`single-slide-editable-pptx`

依赖：M0 的数据契约、离线 OCR 结论和最小工具链。

执行内容：

- 输入校验与微软雅黑预检。
- 离线 OCR、结构化文字块和对象内符号分类。
- 可编辑人工复核文件及校验器。
- 字形 mask、clean plate 候选和非破坏性验证。
- PptxGenJS 合成、比较图和完整中间产物。
- 支持从人工校正后的受影响阶段增量重跑。

验证重点：至少覆盖常规中文、英文/混排、旋转文字、艺术字、容器内文字和对象内符号样例；最终复核后的目标文字全部进入原生文本层。

### M2 评测体系与路线决策

计划子任务：`conversion-evaluation-suite`

依赖：M1 可以稳定产生全部中间产物和结果报告。

执行内容：

- 建立 20～50 页受控评测集和人工标注基准。
- 建立文字发现、内容、分类、mask、clean plate、PPTX 偏差和人工耗时指标。
- 覆盖简体中文、英文、混排、深浅背景、复杂纹理、旋转文字、艺术字、图表和对象内符号。
- 形成继续投入、调整技术路线或收紧支持边界的书面结论。

验证重点：指标可重复计算；失败样例可回溯；不以目标文字位图降级伪造通过。

### M3 多页本地工具

计划子任务：`local-deck-workflow`

依赖：M2 证明核心路线值得继续投入。

执行内容：多页导入、Deck/Slide 模型、本地资产与元数据持久化、阶段任务恢复、批处理、整套 PPTX 导出和 CLI 工作流。

### M4 桌面校正界面

计划子任务：`desktop-review-workbench`

依赖：M3 的稳定领域契约和持久任务模型。

执行内容：Electron + React 桌面壳、页面缩略图、原图/clean plate 对比、文字框编辑、低置信度队列、候选选择和导出操作。

### M5 内容策划与页面生成

计划子任务：`content-and-image-generation`

依赖：M3/M4 已提供稳定的下游转换与人工校正能力。

执行内容：Deck 内容规格、逐页生成、图片 Provider、提示词版本、成本记录和生成结果接入转换引擎。

### M6 可靠性与交付

计划子任务：`local-product-hardening`

依赖：核心工作流已稳定服务开发者本人。

执行内容：异常恢复、备份、日志诊断、macOS 打包、依赖预检、升级迁移和 Windows 兼容性调研。面向外部用户发布需另行立项。

## 4. 质量与评审门

每个子任务必须：

- 有独立 PRD；复杂任务有 `design.md` 和 `implement.md`。
- 在实现前读取相关 Trellis spec；实现后运行 `trellis-check`。
- 验证用户可观察行为和跨阶段数据流，不只验证局部函数。
- 保存失败样例、关键决策和 Provider 研究结果。
- 将已验证的真实约定更新到 `.trellis/spec/`。
- 完成后更新 `ROADMAP.md` 状态，再提交和归档。

## 5. 回滚点

- M0：工具链或离线 OCR 方案不可行时，停在技术研究，不创建大规模产品骨架。
- M1：clean plate 或文本恢复路线不可行时，保留实验和评测产物，回到 Provider/数据契约选择。
- M2：评测结果不足以支撑桌面投入时，暂停 M3/M4，集中改进核心算法或人工校正流程。
- M3 以后：新 UI、持久化或 Provider 不得破坏已冻结的单页产物和数据契约；必要时通过版本迁移兼容。
