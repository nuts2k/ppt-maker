# PPT Maker MVP 路线图

## Goal

为 PPT Maker 建立一份正式、可执行、可持续维护的产品与工程路线图，把现有技术调研转化为按风险排序的里程碑、质量门槛和 Trellis 子任务。路线图应指导项目从核心技术验证逐步走向开发者本人可长期使用的本地工具，避免在图片转可编辑 PPTX 的关键风险尚未验证前过早投入完整 UI 和面向外部用户的产品化建设。

## Background

- 产品目标是生成视觉质量较高、文字仍可在 PowerPoint 中原生编辑的 PPTX，而不是简单地把整页图片放入 PPT。
- 预期核心链路为：内容策划 → 页面规格 → 生成完整设计图 → 识别文字与版面 → 生成去字背景 → 合并原生文本框 → 导出 PPTX。
- `PPT-APP-OPEN-DESIGN-ANALYSIS.md` 已记录初步架构判断和 MVP 验证顺序，但尚未形成带状态、依赖、验收门槛和任务映射的正式路线图。
- `open-design/` 仅为只读参考，不是本项目的代码基础；所有实现必须位于该目录之外。
- 当前代码库尚未形成真实产品代码，因此 `.trellis/spec/backend/` 与 `.trellis/spec/frontend/` 不应被虚构的约定填满，应随着真实实现逐步沉淀。

## Requirements

### R1. 路线图结构

- 路线图必须覆盖从技术验证到桌面产品化的主要阶段。
- 每个里程碑必须说明目标、主要交付物、进入条件、完成条件和明确不做的内容。
- 里程碑必须按风险和依赖排序，而不是按界面或模块目录排序。

### R2. 风险优先级

- 首要验证风险是“图片中的文字能否以可接受质量恢复为 PowerPoint 原生可编辑文本”。
- 完整 Electron UI、多页项目管理和生成式工作流应在单页技术链路与评测标准达到门槛后推进。
- 路线图必须包含明确的继续、降级或停止投入判断点。

### R3. 文本可编辑性硬约束

- 目标范围内的独立版式文字必须 100% 提取并恢复为 PowerPoint 原生可编辑文本，不能以局部或整页位图降级来掩盖文字识别失败。
- 旋转文字必须提取内容，并尽量恢复文本框位置、尺寸、旋转角度、对齐方式和换行结构。
- 艺术字必须提取为原生可编辑文本；阴影、描边、渐变、弯曲、立体等视觉特效不要求首版自动复刻，可由开发者后期人工编辑。
- clean plate 中不得残留本应进入原生文本层的文字；文字删除质量与文字提取完整性必须分别评测。
- “100% 可编辑”指目标文字内容全部进入原生文本层，不等同于所有视觉元素、图表结构或文字特效都必须原生重建。

### R4. 目标文字与对象内符号边界

- 应提取的独立版式文字包括：标题、副标题、说明文字、图注、解释性标签、标注框文字、列表文字，以及用于解释图示含义的文字。
- 与视觉对象融为一体、承担对象细节而非独立说明作用的符号必须保留在图像层中，不应擦除或重复覆盖为文本框。
- 应保留的对象内细节包括：图标细节、象形图、产品或包装标记、图表刻度/标记、仪表盘标记、UI 标记、作为对象组成部分的数字、缩写、货币符号、百分号、装饰符号、字母和其他符号。
- 不得简化图标或删除嵌入对象的细节；页面构图、图标、箭头、容器、背景、阴影、颜色和布局必须保持不变。
- 分类依据是元素在页面中的视觉与语义角色，而不只是字符类别；同样的数字或符号可能在独立说明文字中需要提取，在对象内部则需要保留。

### R5. Clean Plate 非破坏性契约

- 只移除可见的文字字形笔画，不移除承载文字的视觉容器。
- 标题栏、标签栏、丝带、徽章、芯片、卡片、面板、圆角矩形、标注框、标签页、按钮、分隔线、填充、边框、阴影、渐变和装饰性衬底均属于页面结构，必须保留。
- 当文字位于容器内部时，只擦除字符笔画，容器的填充、边框、阴影、尺寸、位置和形状必须保持不变并呈现为空白状态。
- 当文字直接位于页面背景上时，只对文字附近的局部背景进行自然修复。
- 当无法确定某个元素是文字还是文字背景容器时，必须优先保留容器，只移除具有字符特征的笔画。
- 遮罩和修复评测必须分别检查：文字是否彻底移除、容器是否完整保留、非文字区域是否发生误改。

### R6. PPTX 图层与编辑边界

- 首版每页由一张完整 clean plate 背景图和若干 PowerPoint 原生文本框组成。
- 所有目标文字必须位于原生文本层；clean plate 中不得保留这些文字的可见残影。
- 容器、图标、箭头、图表、插画、阴影、渐变和其他非文字视觉元素保留在 clean plate 位图中，不要求转换为 PowerPoint 原生形状、图表或矢量对象。
- 文字容器必须在 clean plate 中视觉完整，但容器本身不要求可编辑。
- 非文字元素的矢量化或结构化重建属于远期独立能力，不作为当前路线图质量闸门。

### R7. 文字内容来源

- 图片中实际可见的目标文字是待恢复内容；生成模型可能在原始文案基础上扩写、改写或新增文字。
- 原始文案不能作为最终文本的唯一真值，只能作为 OCR、视觉理解、分块和语义匹配的参考信息。
- 系统必须能够发现并处理图片中未出现在原始文案里的目标文字，也不能用原始文案静默覆盖图片中的实际文字。
- OCR/Layout、视觉模型和原始文案可以联合提供候选结果，但最终输出必须反映图片中真实可见的目标文字。

### R8. 人工复核与校正闭环

- “100% 目标文字可编辑且内容正确”是最终导出质量目标，不要求自动识别阶段在无人干预时天然达到 100%。
- MVP 必须在导出前提供人工复核与校正能力，至少允许修改文字内容、文本块边界、元素分类、旋转角度、层级和是否参与文字移除。
- 自动流程必须标记低置信度、候选冲突、疑似漏字和“独立文字/对象内符号”分类不确定项，帮助开发者集中复核。
- M1 命令行原型可以通过可读、可编辑、可校验的结构化文件完成复核；桌面阶段再提供可视化校正界面。
- 校正后必须能够从相应阶段重新运行遮罩、clean plate 和 PPTX 合成，而不必重复所有已完成步骤。
- 最终导出前必须能够确认所有目标文字块已经复核，或由开发者显式接受剩余风险。
- 系统应记录每页人工复核与校正耗时，用于判断自动化改进方向，但该耗时不作为路线图阶段的硬性准入门槛。

### R9. 首期用户与使用形态

- 首个可用版本服务开发者本人，是可长期使用、持续迭代的本地工具。
- 首个可长期使用版本聚焦“已有页面图片 → 可编辑 PPTX”，不要求同时完成内容策划和图片生成。
- 输入至少包括一张或多张页面图片；原始文案可以作为可选识别参考，但不是最终文字真值。
- 内容策划、页面规格生成和 GPT Image 等图片生成能力作为后续上游里程碑，在转换引擎稳定后接入。
- 转换引擎必须保持输入来源无关，能够处理由不同模型、工具或人工方式产生的页面图片。
- 优先保证核心生成质量、过程可观察、中间产物可检查、失败可恢复和结果可重复。
- MVP 不要求账号体系、云端同步、多人协作、自动更新、面向普通用户的完整新手引导或商业化能力。
- 在核心链路质量稳定后，可以把“面向外部用户发布”作为后续独立阶段重新评估。

### R10. 本地优先与外部 API 边界

- 项目数据、源图片、中间产物、人工校正记录和导出文件默认全部保存在本机，不依赖自建云端后台。
- OCR 应优先采用可在本机运行的实现，尽量避免仅为文字识别上传整页图片。
- 视觉理解、复杂版面分析或图像编辑能力允许按需调用外部模型 API，但必须通过可替换的 Provider 边界接入。
- 系统必须记录每个阶段使用的 Provider、模型、参数和输入输出，保证结果可追踪和尽量可重复。
- 调用外部服务前应明确展示或记录将发送的数据；API Key 由本地安全存储或运行环境配置管理。
- 完全离线运行可作为后续增强目标，但不是首个可用版本的完成条件。
- 离线 OCR 低置信度时，默认只标记问题区域，不自动把图片发送到云端。
- 开发者可以显式触发云端 OCR/视觉增强；系统应展示将发送的区域或文件及目标 Provider。
- 可以按项目配置受信任 Provider 的自动回退策略，但该能力默认关闭。
- 云端增强必须保留调用范围、模型、结果和是否被人工采纳的记录。

### R11. 首期平台范围

- 首期基准平台为 macOS，使用 Microsoft PowerPoint for Mac 验证导出结果。
- 当前开发机器作为首个可重复验证环境，优先建立稳定的字体、版面和依赖基线。
- Windows 兼容性在核心链路稳定后作为独立里程碑验证，不纳入首期跨平台承诺。
- Linux 不作为 PowerPoint 结果验收平台；是否支持其开发或批处理环境留待后续评估。

### R12. 页面比例

- 首期统一使用 16:9 页面比例，不支持任意自定义比例。
- 输入图片、图像编辑 API 输出、clean plate、比较图和 PPTX 页面必须共享同一 16:9 坐标基准。
- 固定比例用于降低坐标换算、图像修复、字体排版、截图评测和 Provider 能力差异带来的复杂度。
- 其他比例支持属于后续兼容性增强，不作为首个长期可用版本的完成条件。
- MVP 对非 16:9 输入执行显式校验并拒绝处理，不自动裁剪、拉伸或补边。
- 可以容忍由像素取整产生的极小比例误差；容差必须固定、可测试且在处理记录中可见。

### R13. 首期语言范围

- 首期正式支持简体中文、英文以及同一页面或文本块中的中英文混排。
- 支持与正文相关的常见数字、标点、货币和百分比表达；是否提取仍遵守“独立版式文字/对象内符号”的角色边界。
- 字体回退和排版验证以 macOS 与 Microsoft PowerPoint for Mac 为基准。
- 繁体中文、日文、韩文、阿拉伯文、双向排版和其他文字系统不纳入首期验收范围。

### R14. 字体策略

- 首期不识别或猜测源图片中的字体名称，所有生成的 PowerPoint 原生文本统一使用微软雅黑（Microsoft YaHei）。
- 系统仍需还原或允许校正字号、字重、颜色、对齐、行距、换行、边界和旋转角度。
- 运行前必须检查 macOS 上是否可用微软雅黑，并在任务记录中保存实际字体名称和可用性结果。
- 微软雅黑缺失时默认阻止导出并给出明确提示；只有开发者显式配置备用字体后才允许继续。
- 备用字体不得静默启用，导出结果和评测报告必须标明字体偏离基准。
- 不自动下载、安装或嵌入来源不明的字体文件。
- 多字体识别、字体候选匹配和按文本块自动选字体现阶段不纳入首期范围。

### R15. Trellis 任务映射

- 本任务作为路线图父任务，保存整体目标、里程碑关系和跨任务验收标准。
- 可独立验证的交付物应拆成 Trellis 子任务；每个子任务单独规划、实现、检查和归档。
- 依赖关系必须写入任务规划文件，不能只依赖父子目录关系表达。
- 父任务原则上不承载产品代码实现。

### R16. 正式路线图文档

- 在仓库根目录维护 `ROADMAP.md`，作为人和 Agent 都可直接读取的项目级路线图。
- `ROADMAP.md` 应包含里程碑状态、质量门槛、关键决策点和对应 Trellis 任务。
- 完成或调整子任务时，应同步更新路线图状态和必要的范围变化。

### R17. 路线图里程碑

- M0：项目骨架、工具链和最低限度工程约束。
- M1：单页图片转可编辑 PPTX 的端到端命令行原型。
- M2：20～50 页代表性评测集、质量指标、人工校正与支持边界决策。
- M3：多页图片导入、Deck 领域模型、项目持久化和批处理流水线，形成首个可长期使用的图片转 PPTX 工具。
- M4：Electron 桌面壳、任务状态和结果预览/校正界面。
- M5：内容策划、图片生成 Provider 和完整生成工作流。
- M6：可靠性、安装打包、恢复机制和产品化交付。

## Constraints

- 非必要情况下，文档、沟通、注释和提交消息使用中文。
- `open-design/` 下任何文件均不得修改，也不得执行会在其中产生写入的操作。
- 首版不默认采用 Next.js sidecar、通用 Agent CLI runtime、MCP、插件市场、HTML Deck 核心中间格式或通用 `dom-to-pptx` 路线。
- 初步技术方向为 Electron、React + Vite、TypeScript、SQLite、PptxGenJS、本地文件资产和可替换 Provider；正式路线图可以设置验证点，但不能把未经实验验证的选择描述为既成事实。
- 首期只需在 macOS 与 Microsoft PowerPoint for Mac 上建立兼容性基线，不要求同步解决 Windows 差异。
- OCR 必须离线优先；外部 API 是可控的增强或回退路径，而不是默认不可见的强依赖。

## Out of Scope

- 在本父任务中实现产品代码。
- 在路线图规划阶段确定所有页面 UI 细节。
- 复制或迁移 `open-design/` 的代码。
- 在缺少真实项目代码时一次性完成全部 backend/frontend Trellis 规范。
- MVP 阶段的账号体系、云端同步、多人协作、自动更新和商业化能力。
- 首个可长期使用版本中的内容策划和页面图片生成能力。

## Acceptance Criteria

- [ ] 根目录存在经过评审的 `ROADMAP.md`。
- [ ] 路线图中的每个里程碑都有目标、交付物、进入条件、完成门槛和非目标。
- [ ] 路线图明确 M1/M2 对后续桌面端开发的质量闸门。
- [ ] 路线图将目标文字 100% 进入原生文本层设为硬性质量门槛，并把视觉特效复刻与文本可编辑性分开评估。
- [ ] 路线图包含旋转文字和艺术字的专门评测类别。
- [ ] M1/M2 验收覆盖文字移除完整性、容器保真度和非文字区域误改率。
- [ ] M1/M2 能区分独立版式文字和对象内符号，并分别评测漏提取率与误删除率。
- [ ] 原始文案仅作为识别参考，验收覆盖生成图片新增或改写文字的情况。
- [ ] M1 提供结构化人工校正入口，后续桌面阶段提供可视化复核入口。
- [ ] 校正后支持从受影响阶段增量重跑，不要求从头执行完整流水线。
- [ ] 路线图要求记录单页人工复核耗时，但不把耗时设为硬性阶段门槛。
- [ ] 首版 PPTX 的非文字视觉层保持为 clean plate 位图，只有目标文字要求原生可编辑。
- [ ] 输入、处理中间产物和 PPTX 使用统一的 16:9 坐标基准。
- [ ] 评测集覆盖简体中文、英文和中英文混排页面。
- [ ] 首期所有原生文本统一使用微软雅黑，且运行前验证字体可用性。
- [ ] 路线图明确继续、降级或停止路线的关键决策点。
- [ ] 可独立交付的里程碑已经映射为 Trellis 子任务，或明确记录其创建时机。
- [ ] `design.md` 说明产品/技术阶段边界、关键数据流和架构演进原则。
- [ ] `implement.md` 给出创建子任务、维护路线图和开展首个里程碑的执行顺序。
- [ ] 用户完成最终规划评审后，才激活本父任务或首个真正承载实现工作的子任务。
