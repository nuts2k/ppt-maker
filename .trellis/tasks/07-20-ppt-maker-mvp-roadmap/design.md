# PPT Maker MVP 路线图设计

## 1. 设计目标

本设计把 PPT Maker 拆成一组按风险递进的产品阶段。首要目标不是先完成桌面 UI，而是验证并固化以下核心能力：从 16:9 页面图片中识别独立版式文字，保留对象内符号和所有非文字视觉结构，生成 clean plate，再叠加 PowerPoint 原生文本框。

根目录 `ROADMAP.md` 是项目级路线图；本 Trellis 父任务保存路线图的需求、设计、执行计划和子任务关系。具体产品代码由可独立验收的子任务承担。

## 2. 权威性与既有文档关系

- `PPT-APP-OPEN-DESIGN-ANALYSIS.md` 是前期调研和初步判断。
- `ROADMAP.md` 与本任务规划文件记录经过用户确认后的正式决策。
- 两者冲突时，以 `ROADMAP.md` 和本任务 PRD 为准。
- `open-design/` 始终保持只读，只能用于分析和借鉴。

已确认并覆盖早期假设的关键决策：

- 图片中的实际可见文字是识别对象；原始文案仅作参考，不是唯一真值。
- 目标文字不得使用位图降级，最终必须进入原生文本层。
- 对象内数字、缩写和符号按其视觉角色保留，不因形似文字而删除。
- 非文字视觉元素保持为 clean plate 位图，不做矢量化重建。
- 首期页面比例固定为 16:9，字体统一使用微软雅黑。

## 3. 产品阶段边界

```text
技术与工程基线
  → 单页端到端可行性
  → 评测体系与支持边界
  → 多页本地工具
  → 桌面可视化校正
  → 内容策划与图片生成
  → 可靠性、打包与扩展平台
```

每个阶段必须形成可独立验证的交付物。后续阶段不得绕过前一阶段暴露的核心风险；发现基础假设失败时，回到对应阶段修正，而不是用 UI 或位图降级掩盖问题。

## 4. 核心数据流

```text
16:9 source image
  → 输入与字体预检
  → 离线 OCR / Layout 候选
  → 独立文字与对象内符号分类
  → 可选云端视觉增强
  → 结构化 text blocks
  → 人工复核与校正
  → 字形级 mask
  → clean plate 候选
  → 残字、容器和非文字区域验证
  → 选择 clean plate
  → 微软雅黑原生文本框合成
  → PPTX
  → PowerPoint for Mac 结果复核与比较报告
```

原始文案是可选上下文，只能帮助匹配、纠错和判断语义，不得静默覆盖图片中实际可见的扩写或改写内容。

## 5. 核心契约

### 5.1 页面工作区

每页应拥有独立、可恢复的工作目录，至少保存：

- 原始图片与内容哈希。
- 可选原始文案。
- OCR/视觉 Provider 的原始返回。
- 归一化文字块和人工修改记录。
- 文字/对象分类与置信度。
- mask、clean plate 候选和最终选择。
- PPTX、比较图和验证报告。
- 每阶段的 Provider、模型、参数、状态、耗时、错误和版本。

### 5.2 TextBlock

文字块需要表达：

- 图片中实际识别到的内容。
- 位置、尺寸、旋转、阅读顺序和层级。
- 字号、字重、颜色、对齐、行距和换行等可校正样式。
- `layout_text`、`object_integrated_symbol` 或 `uncertain` 分类。
- OCR、视觉模型和原始文案等候选来源。
- 是否参与 mask、是否已人工复核及修改历史。

首期字体不作为识别字段，合成时统一使用微软雅黑；备用字体必须显式配置并进入报告。

### 5.3 Clean plate

mask 只覆盖目标文字的可见字形笔画。容器、边框、阴影、渐变、图标、箭头、图表和对象内符号必须留在图像层中。无法判断时优先保留结构，并把候选标为需要人工复核。

### 5.4 PPTX

每页固定为 16:9：

```text
slide
  ├─ full-slide clean plate image
  └─ editable text boxes (Microsoft YaHei)
```

不创建非文字 PowerPoint 原生形状，不承诺复刻艺术字特效，但保留文字内容、位置和基本排版的人工校正能力。

## 6. 模块边界

首期保持单进程或简单 Worker 结构，避免提前引入独立 daemon。模块通过明确契约隔离：

- `InputValidator`：16:9、图片格式、尺寸和字体预检。
- `LayoutRecognizer`：离线 OCR 优先，输出带坐标候选。
- `VisualClassifier`：区分独立版式文字、对象内符号和不确定项。
- `ReviewStore`：保存结构化人工校正和版本。
- `MaskBuilder`：根据已确认目标文字生成字形级 mask。
- `CleanPlateProvider`：调用本地或外部图像编辑实现并返回候选。
- `SlideValidator`：检查残字、容器损伤和 mask 外变化。
- `PptxComposer`：生成 clean plate + 原生文本框的 PPTX。
- `ArtifactStore`：保存阶段产物、哈希和运行记录。

Electron、SQLite 和独立任务调度在多页与桌面阶段按实际需要引入。M0/M1 应先通过技术研究确定离线 OCR、mask 和 clean plate 的具体实现，不在路线图任务中提前锁死库选择。

## 7. Provider 与本地数据边界

- OCR 默认离线运行。
- 低置信度结果默认只标记，不自动上传。
- 云端增强由开发者显式触发；可以按项目显式开启受信任 Provider 自动回退。
- 图像编辑和复杂视觉理解允许使用外部 API。
- 外部调用必须记录发送范围、Provider、模型、参数、响应和采纳情况。
- 项目数据、资产和校正记录保存在本机，不依赖自建云端服务。

## 8. 状态、恢复与幂等

每个阶段使用显式状态：

```text
pending → running → completed
                  └→ failed / interrupted
```

阶段输入由文件哈希、上游产物版本、配置和 Provider 参数共同确定。输入未变化时允许复用已完成产物；人工修改文字块后，只使 mask、clean plate、验证和 PPTX 等下游阶段失效。

不得因单个阶段失败删除已生成中间产物。多页阶段再把相同模型扩展为可恢复的 Deck 任务。

## 9. 质量模型

必须分别观察：

- 目标文字发现率与最终复核完整率。
- 文字内容正确性、分块、换行、旋转和定位偏差。
- 独立文字与对象内符号的分类错误。
- clean plate 残字、容器损伤和非文字区域变化。
- PPTX 中文字框与源图的视觉偏差。
- 每页人工复核与校正耗时。

最终复核后的目标文字完整率和可编辑率必须为 100%。人工耗时只记录趋势，不作为硬性阶段门槛。M2 建立基线后再决定哪些自动指标需要数值门槛，不在缺少评测数据时虚构阈值。

## 10. 演进与回滚

- M1 证明单页流程不可行时，暂停桌面端投入，优先调整识别、分类或 clean plate 路线。
- 某个 Provider 质量不稳定时，通过适配层替换，不改变核心数据契约。
- 自动识别不足时增加人工校正能力，不允许将目标文字保留为位图作为完成手段。
- Electron 或独立 daemon 只在多页、恢复和交互需求出现后引入。
- 所有阶段保留输入与中间产物，确保可以回退到上一个已验证版本重新生成。
