# M1 单页图片转可编辑 PPTX 原型

## Goal

在 M0 技术基线上跑通一张 16:9 页面图片到 PowerPoint for Mac 可打开的 PPTX 完整链路。最终页面由一张无目标文字残留的 clean plate 背景图和若干微软雅黑原生文本框组成；开发者可通过结构化文件人工复核、校正并从受影响阶段增量重跑。

M1 的价值不是证明自动化可以无人值守达到 100%，而是证明经过明确、可追踪的人工复核后，图片中的全部目标文字都能进入原生可编辑文本层，同时文字容器和对象内符号不被破坏。

## Background

- M0 已建立 Node.js 24、pnpm 10、TypeScript strict/ESM、Apple Vision 离线 OCR、16:9 坐标换算、Zod 契约、微软雅黑预检和 PptxGenJS 输出基线。
- Apple Vision 能离线输出中英文文字、源图像素 bbox 和置信度，但不能可靠提供旋转角度；识别结果可能存在标点和空格误差。
- PowerPoint for Mac 已验证 PptxGenJS 生成的文本框原生可编辑，东亚和 ASCII 字体均可设为 `Microsoft YaHei`。
- 用户已确认原始文案只能作为候选参考，因为生成图片可能扩写、改写或新增文字。
- 用户提供的 clean plate 核心规则是只移除文字字形，保留标题栏、卡片、徽章、按钮、边框、阴影、渐变等文字承载容器；对象内数字、缩写、货币符号、百分号、UI/图表标记等视觉细节不得误删。
- `open-design/` 仅允许只读参考，M1 的全部实现和产物必须位于该目录之外。

## Requirements

### R1. 单页工作区与可重放阶段

- 单次任务输入为一张 16:9 PNG/JPEG，可选提供原始文案参考。
- 为该页建立本地工作区，保存源图、哈希、配置、阶段记录、Provider 信息、版本和所有中间产物。
- 至少拆分为输入校验、OCR 候选、文字块复核、mask、clean plate、PPTX 合成和验证阶段。
- 每个阶段基于输入哈希判断是否可复用；人工校正后只使受影响阶段及其下游失效。
- 阶段失败不得覆盖上一次有效产物，错误和尝试记录必须保留。
- CLI 使用显式阶段命令：`init`、`ocr`、`review`、`assist-review`、`validate-review`、`mask`、`clean`、`pptx`、`report`；阶段名可在设计阶段收敛，但职责边界不得合并消失。
- 提供 `run --from <stage>` 形式的下游重跑入口；不得用一个不可观察、无法在人工校正点暂停的长命令替代阶段工作流。

### R2. 离线优先文字发现

- 默认使用 Apple Vision 离线 OCR，输入不得因低置信度自动上传。
- 输出必须覆盖简体中文、英文和中英混排文字候选，并保存文本、bbox、置信度和可获得的方向信息。
- 必须支持发现原始文案中不存在的图片可见文字。
- 原始文案、离线 OCR 和可选视觉增强结果均作为带来源的候选，不得静默覆盖人工确认值。
- 低置信度、候选冲突、疑似漏字和旋转未知项必须进入复核标记。

### R3. 目标文字与对象内符号分类

- 目标文字包括标题、副标题、正文、列表、图注、解释性标签、标注框文字及用于解释图示的文字。
- 对象内符号包括图标/象形图细节、产品包装标记、图表/仪表/UI 标记，以及作为视觉对象组成部分的数字、缩写、货币和百分号等。
- 分类依据是视觉和语义角色，不是字符类别。
- 不确定项必须显式标记为 `uncertain`，不得默认纳入删除 mask。
- 人工复核必须可以修改分类以及是否参与 mask。
- M1 加入 AI 辅助复核阶段（`slide assist-review`），对 OCR 候选文本做纠错和分类。
- AI 辅助复核使用 OpenAI Responses API、`gpt-5.6-luna`、纯文本输入（不上传图片）和 Structured Outputs。
- AI 辅助复核默认不运行，必须由开发者执行 `slide assist-review --confirm-api` 显式触发；记录发送内容、模型、参数和响应。
- AI 分类为 `layout_text` 或 `object_integrated_symbol` 的块自动设为 `reviewed` 并对应设置 `includeInMask`；分类为 `uncertain` 或带 risk 标记的块保持 `unreviewed`，仍需人工处理。
- 离线 OCR 低置信度或分类不确定不会自动触发 AI 辅助调用。

### R4. 结构化人工校正

- 生成可读、可编辑并可用 Schema 校验的结构化文件，作为 M1 的人工复核入口。
- 至少允许修改文字内容、bbox、旋转角度、换行、分类、层级、是否参与 mask、复核状态和必要的文本样式。
- AI 辅助复核可批量将明确分类的块标记为 `reviewed`；最终导出前，所有目标文字必须为已复核状态，或由开发者逐项显式接受风险。
- M1 不创建 Electron/React 界面；校正通过本地结构化文件和 CLI 完成。
- 记录该页从首次候选到最终确认的人工复核耗时，但不设硬性时长门槛。

### R5. 字形 Mask

- mask 只覆盖已确认目标文字的可见字形笔画，不覆盖承载文字的容器。
- 旋转文字和艺术字的内容必须进入文本层；其字形也必须从背景中移除。
- 对象内符号和不确定项默认不进入 mask。
- mask 生成必须保留与源图一致的像素尺寸和 16:9 坐标基准。
- 必须保存可检查的 mask 预览或叠加图，使开发者能发现漏字、膨胀过度和误覆盖。
- mask PNG 必须完全由系统根据已复核的结构化文字块和版本化算法生成，开发者不得直接涂改、替换或导入人工 mask 文件。
- 人工修正只能发生在结构化数据层，包括文字块边界、字符/子区域、旋转、分类、是否参与 mask，以及设计中明确暴露的 mask 生成参数；修改后必须重新运行 mask 阶段。
- mask 阶段必须验证输出来源和输入哈希，拒绝来源不明或被外部修改的 PNG 继续进入 clean plate 阶段。

### R6. Clean Plate（固定 GPT Image 2）

- clean plate 必须保持页面构图、容器、图标、箭头、图表、插画、阴影、颜色、渐变和非文字细节不变。
- 容器内文字被移除后，容器的填充、边框、阴影、尺寸、位置和形状必须保持完整。
- 直接位于背景上的文字只修复局部背景。
- clean plate 唯一生成路径为 OpenAI Image API 的 `/v1/images/edits`，模型固定为 `gpt-image-2`；M1 不实现其他 Provider，也不允许导入人工处理的 clean plate 作为回退。
- 请求必须同时提交源图、带 alpha 通道且与源图同尺寸/格式的 mask，以及版本化 clean plate 提示词。
- `gpt-image-2` 的 mask 是模型引导而非逐像素硬约束，因此输出必须经过文字残留、容器完整性和非文字误改检查；质量不合格时只能调整文字块、mask、提示词或允许参数后重新调用。
- `gpt-image-2` 自动以高保真方式处理输入图片，M1 不设置 `input_fidelity`。
- M1 固定请求 `size: "2048x1152"`、`quality: "high"`、`output_format: "png"`，不暴露其他尺寸或质量档位。
- 外部调用必须由开发者显式触发，并记录发送的文件、`gpt-image-2` 模型字符串、尺寸、质量、格式、提示词版本、请求标识、耗时、用量、错误和结果哈希。
- API 失败或质量不足时保留原始输入、mask、失败输出和尝试记录，不得跳过 clean plate 阶段继续导出，也不得人工替换产物绕过流程。

### R7. 原生文本层与 PPTX

- PPTX 页面固定为 wide 16:9，以 clean plate 作为全页背景。
- 所有确认目标文字生成 PowerPoint 原生文本框，统一使用 `Microsoft YaHei`。
- 还原或允许人工校正文本框位置、尺寸、旋转、字号、字重、颜色、对齐、行距和换行。
- 艺术字的阴影、描边、渐变、弯曲和立体效果不要求自动复刻，内容必须可编辑。
- 微软雅黑不可用时默认阻止导出；显式备用字体必须记录在报告中。

### R8. 验证与报告

- 生成源图、clean plate、最终 PPTX 渲染结果的可比较产物。
- 分别记录文字发现/内容、分类、mask、clean plate、文本框排版和可编辑性检查结果，不用单一相似度分数代替。
- 至少验证常规中文、英文/混排、旋转文字、艺术字、容器内文字和对象内符号场景。
- 自动检查不得冒充 PowerPoint 人工验证；最终样例必须在 Microsoft PowerPoint for Mac 中确认可打开和可编辑。
- 验收使用双样例：仓库内可公开提交的合成 fixture 用于稳定自动化测试；另使用一张开发者实际生成的复杂页面作为端到端主验收样例。
- 真实页面至少包含容器内文字、对象内符号，以及旋转文字或艺术字中的一种；真实页面及其衍生产物是否提交到 Git 由素材敏感性决定，但本地工作区必须保留完整验证记录。
- clean plate 和最终 PPTX 分别设置人工质量门：clean plate 未在结构化复核记录中显式标记为已接受时，不得生成正式 PPTX；PPTX 未在 PowerPoint for Mac 检查并标记为已接受时，该页不得报告为完成。
- 每次人工接受必须记录时间、接受者、对应产物哈希和备注；上游输入变化会自动使下游接受状态失效。

## Acceptance Criteria

- [ ] 一条有文档的 CLI 工作流能从单张 16:9 PNG/JPEG 创建本地页面工作区并产出全部阶段文件。
- [ ] 默认离线 OCR，低置信度不会自动上传，候选来源与人工采纳结果可追踪。
- [ ] AI 辅助复核（`assist-review`）只能显式触发，能对 OCR 文本做纠错和分类，明确分类的块自动标记为 reviewed，并记录完整调用元数据。
- [ ] 可编辑结构化文件能通过 Schema 校验，并支持文字内容、bbox、旋转、分类、mask 参与和样式校正。
- [ ] 人工校正后只重跑受影响阶段及其下游，未失效的阶段产物可复用。
- [ ] 目标文字和对象内符号可被区分；不确定项不会默认删除。
- [ ] mask 只能由系统从已校验结构化数据生成；人工修改或替换 mask PNG 会被完整性校验拒绝。
- [ ] mask 与 `gpt-image-2` 的 `2048x1152`、`high`、PNG clean plate 均可单独检查，且 clean plate 中目标文字无可见残留。
- [ ] 标题栏、卡片、徽章、按钮、边框、阴影、渐变等文字容器保持完整。
- [ ] 图标、图表、仪表、UI、包装等对象内数字和符号未被误删或重复覆盖。
- [ ] 旋转文字和艺术字内容进入微软雅黑原生文本框，特效允许后续人工补齐。
- [ ] 最终复核后的全部目标文字内容正确并位于原生可编辑文本层。
- [ ] PPTX 在 Microsoft PowerPoint for Mac 中为 16:9、可打开、文本可编辑，字体预检和偏离记录符合约定。
- [ ] 生成分阶段验证报告并记录人工复核耗时，但耗时不作为硬门槛。
- [ ] 合成 fixture 的自动化链路可重复运行，真实复杂页面完成一次完整端到端人工验收。
- [ ] clean plate 与 PPTX 两个人工质量门均基于产物哈希生效，上游变化后旧接受记录不能继续放行。
- [ ] 格式检查、类型检查、测试和构建全部通过，`open-design/` 无改动。

## Out of Scope

- 多页 Deck、批处理、SQLite 持久化和应用重启恢复。
- Electron/React 可视化复核界面。
- 内容策划、页面规格生成和页面图片生成。
- 非文字元素矢量化、原生 PowerPoint 图表/形状重建。
- 艺术字复杂特效的自动复刻和原字体识别。
- 完全无人值守达到 100% 或将目标文字降级保留为位图。
- 其他图像编辑 Provider，以及人工导入或替换 clean plate 的回退路径。
- Windows/Linux 正式支持。
