# PPT Maker 路线图

> 状态：执行中（M0 已完成；M1 已完成；下一步 M2 评测体系）
>
> 目标用户：开发者本人
>
> 首期平台：macOS + Microsoft PowerPoint for Mac
>
> 首期产品：已有 16:9 页面图片 → 文字原生可编辑的 PPTX

## 1. 产品目标

PPT Maker 的核心价值是在保留 AI 生成页面视觉效果的同时，把独立版式文字恢复为 PowerPoint 原生可编辑文本。

首版每页采用：

```text
完整 clean plate 背景图
+
微软雅黑原生文本框
```

容器、图标、箭头、图表、插画、阴影和渐变保留在背景图中，不要求原生化。标题、正文、图注、解释性标签、列表和标注文字必须进入文本层；与视觉对象融为一体的数字、缩写和符号必须保留为对象细节。

最终经过人工复核后，目标文字的内容正确率、覆盖率和可编辑率必须达到 100%。文字特效可以后期人工补齐，不以位图降级代替目标文字。

## 2. 已确认的产品约束

- 服务开发者本人长期本地使用，暂不面向外部用户发布。
- 首个可用产品聚焦图片转 PPTX，内容策划和图片生成后续接入。
- 输入和输出统一为 16:9；非 16:9 输入默认拒绝。
- 首期支持简体中文、英文和中英混排。
- 所有原生文本统一使用微软雅黑；缺失时默认阻止导出，可显式配置备用字体。
- OCR 离线优先；低置信度结果默认不自动上传。
- 云端 OCR/视觉增强由开发者显式触发，可按项目配置受信任 Provider 自动回退。
- 图像编辑和复杂视觉理解允许调用外部 API，项目数据和中间产物保存在本机。
- 原始文案仅作识别参考，图片中实际可见的独立版式文字才是恢复对象。
- 必须支持人工复核、结构化校正和受影响阶段增量重跑。

## 3. 总体质量原则

分别评测以下维度，不能用单一“视觉相似”分数代替：

- 目标文字发现、内容、分块、换行、旋转和位置。
- 独立版式文字与对象内符号的分类。
- clean plate 的文字残留、容器完整性和非文字区域误改。
- PowerPoint for Mac 中的文本框排版和最终可编辑性。
- 每页人工复核与校正耗时；只记录趋势，不设硬性准入门槛。

### M1 单页图片转可编辑 PPTX 原型

状态：已完成（2026-07-21）

目标：跑通单张图片从识别、人工校正、去字到 PPTX 的完整链路。

进入条件：M0 完成核心数据契约和技术基线。

已交付（合成 fixture 自动化全绿）：

- 单页工作区 + 阶段 DAG + 输入指纹复用/失效；显式阶段命令与 `slide run --from`（上传门/人工门停止）。
- 16:9 校验、微软雅黑预检、Apple Vision 离线 OCR（含字符/子串 `glyphHints`）。
- 显式云端视觉分析（Responses API `gpt-5.6-sol`），候选合并生成可编辑 `text-blocks.json`，`validate-review` 结构化校验。
- 自动字形 mask（sharp I/O + TS 像素算法：颜色/边缘/连通域/膨胀/多边形/glyphHints 软先验）与覆盖率统计。
- gpt-image-2 clean plate（固定 2048x1152/high/PNG）+ 离线检查（尺寸/文字残留/mask 外差异/容器环）+ 人工 accept 门。
- 微软雅黑原生文本框 PPTX + ZIP/XML/16:9/文字/字体/形状自动检查 + PowerPoint for Mac 人工 accept 门。
- 分阶段报告（自动检查与人工接受分区，未完成不为成功）；合成 fixture 五类元素端到端覆盖。

技术结论：

- Apple Vision 可离线输出中英文与字符四边形；`glyphHints` 仅作 mask 局部先验，不是精确字形轮廓。
- sharp（预编译平台包，无 install 脚本，免白名单）负责编解码/裁剪/合成，像素算法用 TS 自实现，字节级确定、可建像素基线。
- 指纹投影（design §6）：mask/clean 只对几何/分类/mask 参数投影哈希，内容/样式变更只重跑 PPTX。
- gpt-image-2 类型契约在 openai@6.48.0 完整可用；`size` 经 `(string & {})` 软收口，固定常量 + 像素尺寸断言补偿；mask 为模型引导非硬约束，故辅以离线残留/容器检查 + 人工门。

进入 M2 条件：✅ 已满足 — 开发者实际生成的复杂真实页完成完整本地工作区流程，人工确认全部目标文字进入原生层、clean plate 无残留、容器与对象内符号未破坏，并在 PowerPoint for Mac 确认 16:9/可打开/可编辑/微软雅黑。

非目标：多页批处理、桌面 UI、内容生成、非文字元素矢量化。

Trellis 子任务：`07-20-single-slide-editable-pptx`

### M2 评测体系与路线决策

状态：待开始

目标：用真实样例量化核心链路，决定是否以及如何进入长期工具建设。

进入条件：M1 能保存并重放全部中间产物。

主要交付物：

- 20～50 页受控评测集与人工标注基准。
- 中文、英文、混排、深浅背景、复杂纹理、旋转文字、艺术字、图表和对象内符号分类样例。
- 文字、分类、mask、clean plate、PPTX 和人工耗时指标。
- 失败模式目录与继续投入、调整技术路线或收紧支持边界的决策记录。

完成条件：评测可重复执行；失败可以定位到具体阶段；形成下一阶段的书面决策。

非目标：通过目标文字位图降级提高表面成功率；提前开发完整桌面应用。

计划 Trellis 子任务：`conversion-evaluation-suite`

## 5. 关键决策点

### M1 后

- 若目标文字无法可靠分类或 clean plate 大量破坏容器，暂停 UI 投入，调整识别、mask 或图像编辑路线。
- 不允许通过把目标文字保留在背景图中宣称完成。

### M2 后

- 根据评测结果决定继续 M3、增加技术实验，或收紧明确支持的页面类型。
- 人工校正耗时用于排序自动化改进，不单独决定项目是否继续。

### M3/M4 后

- 只有图片转 PPTX 工具已经稳定可用，才接入内容策划和图片生成。
- 面向外部用户、Windows 正式支持和商业化能力必须单独立项。

## 6. Trellis 使用方式

- 本路线图对应父任务：`07-20-ppt-maker-mvp-roadmap`。
- 每个里程碑由独立子任务负责规划、实现、检查和归档。
- 后续子任务在前一里程碑接近完成时创建，避免一次性建立大量过时计划。
- 完成子任务后先更新本文件状态和关键决策，再提交并归档任务。
- 父任务在主要路线图工作完成前保持可追踪，不提前归档。

## 7. 明确不采用的首期方向

- 修改或直接继承 `open-design/`。
- Next.js sidecar、通用 Agent CLI runtime、MCP 和插件市场。
- HTML Deck 作为核心中间格式。
- 通用 `dom-to-pptx` 作为首版导出引擎。
- 任意比例、多字体识别、非文字元素矢量化和完全离线模型体系。
- 账号、云同步、多人协作、自动更新和商业化能力。
